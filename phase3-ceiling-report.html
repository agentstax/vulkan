TL;DR

throughput = min(  supply(batch)  ,  ack_capacity(workers)  )

Two independent limits (once the claim query is indexed). Which one binds depends on how big your batch is. Small batch → supply-bound. Big batch → ack-bound. There is no fixed "single prefetcher" ceiling.

  • Ceiling on this box: ~20–22k msgs/sec at 64 workers — and it's the ack path (one single-row UPDATE+commit per message), not the claim loop.
  • Claim cost is sublinear in batch (~3.4 µs/row asymptote), so a single prefetch loop can supply ~290k rows/s. You don't need parallel prefetchers anywhere near current scale — a bigger batch lifts supply out of the way.
  • At batch=100 the system is co-limited (~17–18k): per-claim round-trip caps supply right around where the ack path saturates. Raising batch reveals the ack ceiling (~22k), which then scales with workers.
  • The index still matters most. Without it, the ORDER BY id claim scans the graveyard of done rows — one claim went 0.057 ms → 41.8 ms (730×). That collapses the whole supply curve. Fixed in migration 005.
  • Real dip at 2 workers — the group-commit valley.

1 · The model: min(supply, ack)

Three things can limit a single consumer. Whichever is lowest wins:

Claim query cost

How long one claim takes. Cheap with the right index; O(graveyard) without it. Set by indexing, not tuning.

Prefetch supply

batch / L(batch). At small batch dominated by the ~6 ms round-trip. Scales with batch → ~290k/s.

Worker ack throughput

One single-row UPDATE+commit per message, group-commit-limited. Scales with concurrency.

3 · Throughput vs. batch — supply lifts, then the ack path caps it

64 workers fixed, 200k backlog, batch swept 100→50,000. Throughput rises ~17.7k→~22.6k then flattens: by batch=50,000 the prefetcher does just 4 claims for the whole run (~0.8 s of a 10 s run) — it's idle most of the time. The remaining wall is the ack path.

4 · The claim-cost leg: the index that keeps it cheap

Everything above assumes the claim stays cheap. It only does with the right index. The claim does ORDER BY id, so the planner walks the primary key and filters status inline. As done rows pile up at low ids, every claim scans through the graveyard to reach live rows:

Without the index a claim degrades 0.057 ms → 41.8 ms (730×) as the queue drains — L(batch) stops being cheap-per-row and the entire supply curve collapses (we measured ~6.4k/s at 64 workers in this state, with backends 87.5% idle on ClientRead — genuinely claim-bound). This is the one regime where the prefetch loop really is the bottleneck, and it's a missing-index problem, not a structural one.

The fix (migration 005, applied): a partial index keyed on id covering both live states, so the ordered scan never touches the graveyard and FIFO ordering is preserved. The existing idx_claim_ready / idx_claim_processing (004) can't drive an id ordering across the ready OR processing predicate, so this query never used them.

CREATE INDEX idx_claim_active ON message_log (id) WHERE status IN ('ready','processing');

5 · The 2-worker valley

In the low-concurrency series every 2-worker trial came in below the 1-worker best (1w ≈ 1,700/s; 2w clustered 1,070–1,180/s). This is the group-commit valley, a property of the ack path under synchronous_commit = on:

  • 1 worker: commits are serial — one WAL fsync per ack, no cross-backend contention.
  • 2 workers: two backends now contend on the commit path, but there still aren't enough concurrent committers for Postgres to group-commit (batch many transactions into one fsync). You pay the contention without the amortization → a net regression.
  • 4+ workers: each fsync flushes several transactions' WAL at once; the cost amortizes and throughput climbs steeply.

This is also why the ack ceiling scales with workers — and why batched acks (below) would raise it: fewer, fatter commits.

6 · How to go faster (in order)

  1. Keep claims cheap — the 005 index (done). Non-negotiable; without it nothing else matters as the queue fills.
  2. Raise batch + buffer to clear the supply limit. Cheap, single-loop, gets you to the ack ceiling. Bounded for real jobs by the lease window (every buffered row's lease_until is ticking — the shallow-buffer rule is a lease-safety constraint, not a throughput one).
  3. Add workers to raise the ack ceiling. Throughput scaled 8k→20k across 8→64 workers; bounded by max_connections.
  4. Batch the acks. The ack path is the wall — one single-row commit per message. Grouping acks (multi-row UPDATE per commit) cuts per-message fsync + round-trip, the same amortization trick applied to the write side. Costs at-least-once granularity.
  5. Multiple prefetchers (the queue.go broadcast TODO) — only needed past ~290k/s supply, i.e. far beyond the current ack-bound regime. Not the near-term lever.
  6. Archive done/dead rows — keeps the live table small so claim cost stays flat (complements the index from the other side; Phase 3.5).
