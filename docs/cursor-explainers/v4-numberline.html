The Managed Cursor

A precise, data-structure-first view — watermark & ranges. No metaphor, just the literal thing rendered cleanly. For implementers who want to build it.

A consumer group reads a durable, append-only log of messages with strictly-increasing offsets. Work finishes out of order. We track group progress with two pieces of state: one integer watermark (everything at or below it is fully settled) and a small set of ACK ranges above it (the messy window of mixed outcomes). This is exactly TCP cumulative‑ACK + SACK, and Pulsar's managed‑cursor mark‑delete position + individually‑acked ranges.

Why this beats a naive single cursor. A naive design advances one position only while the very next offset is done. When offset 1 goes DEAD, that cursor freezes at 0 forever — head‑of‑line blocking. The managed cursor treats dead as resolved, so the watermark slides past it (the dead row is kept in the dead‑letter store for audit).

Where this model leaks

This rendering is deliberately literal, but even "no metaphor" hides things. Honest accounting:

What it makes clear

  • The contiguity rule. The watermark only moves over a gapless run; the shaded zone makes "no holes below me" visually obvious.
  • Cost model. One int + a range set that grows with disorder, not with the log length. You can see the table stay small.
  • Roll-up cascade. Filling one gap can let the watermark jump several spots in a single slide.
  • No head-of-line blocking. Dead counts as resolved, so one poison message can't freeze the group.
  • GC. The 1:1 link between "watermark passed it" and "row is deletable" is explicit.

What it hides / distorts

  • Concurrency. The stepper looks like one ordered actor. Really, many uncoordinated workers claim distinct offsets and ack independently; the watermark is advanced lazily by a separate background roll-up and is usually stale-low.
  • Time & leases. "In-flight" hides lease expiry, retries with backoff, and the at-least-once redelivery that can re-fire a message.
  • Crash recovery. The clean slide hides that un-flushed roll-up progress can be lost; the watermark is then merely recomputed (harmlessly) on restart.
  • 1-D tidiness. A real range set can be far more fragmented than 8 cells, and merges/splits churn under load. The neat bars understate the bookkeeping.
  • "Done" vs "dead" look equal. Both are "resolved" for the watermark, but operationally a dead-letter pile is an incident, not a success.
Fidelity footnote. This is precisely TCP cumulative ACK + selective ACK (SACK) blocks (RFC 2018) and Apache Pulsar's managed‑cursor model: a mark‑delete position (our watermark) plus individuallyDeletedMessages ranges (our ACK ranges). The hot path (claim + ack) only touches per‑message rows; the watermark is advanced off the hot path. Guarantee: at‑least‑once.
