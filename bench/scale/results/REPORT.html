How should Vulkan fan out an event log to consumers?

A head-to-head benchmark of 5 Postgres designs · throwaway Postgres 18.4 (OrbStack, 8 cores / 8 GB), pgbench/psql 17.9 · ~150 measured runs, audited twice & corrected · 2026-06-21 · jump to the second-audit corrections →

The one-sentence answer: writing a deliveries row per (consumer-group, event) — the obvious "queue table" approach — costs a row write on both the produce side and the consume side, and the consume side is the wall (~124k work-items/s). Designs that skip the per-event row (claim straight from the log, advance a cursor per batch) run ~3–4× faster at matched scale and ~8× as the number of consumer groups grows, and sustain ~4–8× more end-to-end load. The richness a deliveries row buys you (independent/out-of-order acks, retries, dead-lettering, per-message visibility) is exactly what makes it expensive — and most messages never need it.

1 · Terms & how to read the numbers

Every number in this report is one of a few specific things. Pinning these down first prevents the conflation that plagues throughput talk.

TermDefinition
eventOne record appended to the immutable log (a row in events).
consumer groupAn independent subscriber that must see every event (e.g. "billing", "search-index"). G = number of groups.
unit (work-item)One (group, event) pair — one atomic piece of work. One event with G groups = G units.
units/s primary metricWork-items completed (claimed + acked) per second. This is what the database actually does, so it's the honest measure of capacity.
events/s fully processed= units/s ÷ G. An event is "fully processed" only once all G groups have acked it. This is the user-facing view.
append rateEvents written to the log per second (produce side only).
breaking pointThe highest offered append rate at which the backlog stays bounded (consumers keep up). Past it, the backlog grows forever.
Worked example. d4 at saturation = 287k units/s with G=10 groups. That's 287,000 ÷ 10 ≈ 28.7k events/s fully processed — every one of those events got handled by all 10 groups.

2 · The mental model: a 3-stage pipeline

Every design is some arrangement of three stages. Where a design puts the cost (or skips a stage) is the entire story — so the results are organized stage-by-stage, not design-by-design.

A · Append

Producer writes the event to the log. Cost is dominated by the commit fsync (durability), which batching amortizes.

B · Fanout / materialize

Turn 1 event into G work-items — one deliveries row per group. This is the optional, contentious stage. Two designs skip it entirely.

C · Consume

A worker claims a work-item, processes it, and acks. Cost depends entirely on whether "a work-item" is a row to rewrite or just a cursor to advance.

How each design maps onto the pipeline (struck-through = stage skipped):

DesignAppendFanoutConsume
d1 sync trigger▣ fused — fanout runs inside the append transaction (a trigger)per-row claim+ack
d2 NOTIFY+projectorbare appendasync (separate process, woken by NOTIFY)per-row claim+ack
d3 polling projectorbare appendasync (separate process, polls)per-row claim+ack
d4 claim-from-logbare appendnoneclaim a range from the log, advance cursor
d5 range-leasebare appendnoneclaim range + write 1 lease row, advance cursor
Why d1/d2/d3 share a "consume" number but nothing else. All three end up with the identical per-(group,event) deliveries row, so they run the same claim+ack code and have the same consume ceiling (~124k units/s). They differ entirely in stage B: d1 pays fanout synchronously on the producer's commit; d2/d3 pay it asynchronously in a separate process. That difference dominates their end-to-end behavior — which is why they are not interchangeable despite the shared consume cost.

3 · The five designs

d1 Synchronous trigger fanout

An AFTER INSERT statement-level trigger inserts one deliveries row per group, in the producer's own transaction. Simple, strongly consistent: the moment an event commits, its work-items exist.

Pays: fanout latency on every append; per-row claim+ack on consume.

d2 NOTIFY + async projector

Producer commits only the event + a coalesced pg_notify (once per statement, never per row). A separate "projector" process LISTENs and materializes the deliveries rows off the hot path.

Pays: fanout asynchronously (low producer latency); per-row claim+ack on consume.

d3 Polling projector

Same as d2 but the projector polls (WHERE offset > projected) instead of waiting on a notification. Simpler, no LISTEN plumbing; trades a little latency.

Pays: same as d2.

d4 Claim-from-log

No per-event rows at all. Each group has a claim frontier (a cursor). A worker atomically bumps it (“I took offsets 5000–5200 for group g”), reads those events, processes, and commits the new cursor. Rows are written only for exceptions (retries/dead-letters).

Pays: one cursor update per batch + a per-event read. No fanout.

d5 Range-lease

Like d4, but each claimed range also writes a durable lease row (deleted on commit). That lease makes a crashed range reclaimable — Kafka-style offset ownership.

Pays: d4's cost + one lease INSERT/DELETE per batch (~3%).

The thing under test

d1/d2/d3 answer “what's the best way to materialize a row per work-item?” d4/d5 answer the deeper question: “do we need the row at all?” They are the ceiling the others are measured against.

4 · How it was measured

  • Steady-state, not peak. Each run samples throughput once per second; we drop a warm-up window and take the median (with p10/p90) of the remaining 20–25s. Drain ceilings are the median of 3 independent repeats.
  • Real work only. Throughput is a per-second delta of a completion counter that only the ack path increments — so empty/no-op claims can't inflate it.
  • Correctness checked every run. Fanout must produce exactly events × groups rows; claim-from-log must complete exactly events × groups units with no double-processing and no gaps (verified to the row under 32 concurrent workers).
  • Breaking point = ramp the offered append rate until the backlog (head − processed) stops being flat and starts growing without bound.
  • Independently audited. A separate adversarial review re-checked the harness and forced two corrections (see §10) — the most important being that the no-fanout read had to actually touch the event payload, not just count an index.

6 · Stage B — Fanout / materialize

This stage exists only for d1/d2/d3 (d4/d5 skip it). It writes one deliveries row per group per event. The question: do it synchronously in the trigger (d1), or asynchronously in a projector (d2/d3)?

d1 — synchronous trigger: the fanout tax

Because the trigger runs inside the producer's transaction, the producer's effective append rate collapses as groups grow — it has to write G delivery rows before its own insert commits.

groups (G)d1 append (events/s)delivery-rows/s written
1~19k19k

Why: the trigger's row-writing throughput plateaus at ~0.5M delivery-rows/s on this box, no matter the group count, sync setting, or batch size — it's bound by writing rows, not by fsync. So the producer's event rate is simply ~0.5M ÷ G. At 500 groups the producer is choked to under 1k events/s.

d2/d3 — async projector: decouple fanout from the commit

The projector materializes the same rows in a separate process, so the producer commits at the bare rate. Its own ceiling:

projector config (G=10)delivery-rows/snote
1 process, small batch (1k offsets)290k
1 process, batch 20k offsets342kbatch plateaus ~20k

8 processes629kregresses — core oversubscription
Why: the projector's set-based INSERT…SELECT hits roughly the same ~0.5–0.7M rows/s wall as the trigger — materialization costs the same no matter who does it. The win isn't speed, it's decoupling: the producer's commit no longer waits for fanout. NOTIFY (d2) and polling (d3) reached the same throughput; NOTIFY just has lower idle latency. But see §8 — that decoupling has a sting when everything runs at once.

7 · Stage C — Consume / drain

This is the decisive stage. Numbers below are the corrected, replicated drain ceilings (median of 3 reps, G=10, batch 200, with a real per-event heap read on d4/d5).

Designunits/s (cc16)units/s (cc32)vs per-row
d1d2d3 per-row claim+ack124k99k1×
d4 claim-from-log481k430k3.9–4.3×
d5 range-lease400k411k3.2–4.1×
d4 claim-from-log @ G=100768k7.7×

Why the per-row path is the wall (~124k units/s)

Completing one unit means two UPDATEs: ready→inflight (claim) then inflight→acked (ack). Each UPDATE writes a new heap row-version and maintains the index. So it's ~2 row-writes + index upkeep per unit, and it's bound by row-write throughput, not commit fsync. The evidence that it's row-bound, not fsync-bound:
  • synchronous_commit=off gave ~no improvement (112k vs 116k) — removing the fsync didn't help, because fsync wasn't the limit.
  • Collapsing claim+ack into one transaction made it worse (51k) — one fewer commit, but it holds the row locks longer, so 32 concurrent workers hit far more SKIP LOCKED contention.
  • More clients past cc16 regresses (99k at cc32) — lock/latch contention, not throughput headroom.
The only lever that helped was batching the claim (b1000 → +15%).

Why claim-from-log is 3–8× cheaper

A claim-from-log worker writes nothing per event. It atomically bumps one cursor row ("took offsets 5000–5200 for group g") and reads those events. The only write is the cursor advance — once per batch of 200, i.e. ~1/200th the writes. It still reads each event (a heap fetch — we forced that with count(payload) so the comparison is fair), but a read is far cheaper than two writes + index maintenance.
Why it scales with group count (the 4× → 8× jump): the cursor is per group.
  • G=1, 32 workers: all 32 fight over one cursor row → serialized → ~100–128k units/s. This is the single-frontier contention wall.
  • G=100, 32 workers: the 32 spread across 100 cursor rows → almost no contention → 768k.
This is the exact opposite of materialize designs, where more groups means more rows to write (slower). For claim-from-log, more groups means less contention (faster).

d5 vs d4: range-lease is consistently a touch below claim-from-log — the extra INSERT+DELETE of the lease row per batch. You buy crash-safe range reclaim for ~3–5%.

8 · Stage D — End-to-end & breaking points

Now everything runs at once: producer + (projector) + consumers, G=10, on the same 8 cores. This is where the stages interact — and where d2's async decoupling backfires.

Designappend (ev/s)saturation (units/s)events/s fullybreaking point10M-row spill
d1 sync trigger19.5k75k7.5k~5–10k−26%
d2 NOTIFY+proj3.4k34k3.4k~3.4k (capped)—
d3 poll proj26.9k43k4.3k~5–10k—
d4 claim-from-log68.7k287k28.7k~20–40k−6%
d5 range-lease72.8k300k30k~20–40k−14%

Breaking point = highest offered events/s before the backlog diverges. Spill = throughput change when the hot table is pre-grown to 10M rows.

The counter-intuitive finding: async fanout (d2) was the slowest end-to-end — worse than the synchronous trigger.
d2 saturates at 34k units/s vs d1's 75k. Reason: 8 cores, and at saturation you're running the producer + 4 projector processes + 16 consumer threads + a NOTIFY on every insert statement, all flat-out. The projectors and NOTIFY machinery steal CPU and WAL bandwidth, starving the producer to ~3.4k ev/s — so little arrives that nothing else can run fast either. Async fanout decouples latency (the producer's commit doesn't wait for fanout) but not throughput — everyone still shares the same cores. On a box with spare cores this would look very different; under saturation on a fixed core budget, moving fanout to its own process just adds contention.
Why d4/d5 also degrade least with table size (−6% vs d1's −26% at 10M rows): claiming a contiguous offset range is a clean B-tree range scan that stays cheap as the log grows. The per-row path claims scattered ready rows via a partial index over a table that keeps accumulating acked rows, so it feels table growth more. (Caveat: 10M rows ≈ 1.5GB still fit cache, so this is an index/table-size effect, not true disk spill — see §12.)

9 · Optimization ladders — what helped, what didn't

Append

✓ batching (biggest lever, ~40–70×)
✓ sync=off (~2–3.7×)
✓ commit_delay=50µs (+35%)
✓ UNLOGGED (ceiling 1.45M)

Materialize (projector)

✓ parallel shards → 4 cores (~2×)
✓ offset-window batch ≈ 20k

✗ >4 shards regresses (oversubscription)

Per-row consume

✓ batch the claim (+15%)

✗ sync=off (no help — row-bound)
✗ combined 1-txn claim+ack (worse)
✗ >16 clients (regresses)

Claim-from-log consume

✓ more groups (less frontier contention)
✓ bigger batch

✗ many clients on few groups (single-frontier wall ~100–128k)

↻ unexhausted: shard the frontier (multiple cursor rows/group) — would lift the low-group wall.

Range-lease consume

= claim-from-log − ~3–5% (lease row write/delete).

Key takeaway

The two halves have different bottlenecks: append is fsync-bound (fix with batching), per-row consume is row-write-bound (no fsync trick helps). That asymmetry is why the per-row path can't be tuned out of its ~124k wall.

10 · The correction story (why these numbers are trustworthy)

An independent adversarial audit reviewed the harness and the raw data. It confirmed the correctness invariants hold exactly, but caught two issues that inflated the first-pass headline from a real ~3–8× to a misleading 40×. Both are fixed in the numbers above.

Fix 1 — the no-fanout read was cheating. The claim-from-log "read the events" step was SELECT count(*), which Postgres answered with an index-only scan on the offset primary key — it never touched the heap or read the event payload. So d4/d5 were "completing" work without doing a real consumer's read. Fix: changed to count(payload) (payload isn't indexed, so it forces a per-event heap fetch). This slowed d4/d5 (saturation 480k→287k) and is why the matched advantage is ~3–4×, not the inflated number.
Fix 2 — exhausted backlogs & N=1. Several top figures were measured on backlogs that fully drained mid-window (so the trailing samples were idle, distorting the median), and most points were single runs with ~20% run-to-run variance. Fix: re-ran the load-bearing drains with non-exhausting backlogs and 3 repeats each, reporting median-of-medians. The clean "141k" per-row number I'd cited turned out to be one unreplicated outlier; the honest replicated peak is 124k.

What the audit confirmed as solid: the fanout count invariant (deliveries == events×groups, exact on every run), the no-loss/no-dup frontier (done == events×groups exact under 32 concurrent workers), the empty-claim-proof rate metric, and the overall ranking.

12 · Trust rating & caveats

Direction & matched ratios — HIGH

Replicated (3 reps), fair per-event reads on both sides, non-exhausting backlogs, and exact correctness invariants checked every run. The ranking and the ~3–8× advantage are solid.

Absolute magnitudes — MEDIUM

Single box, 8 usable cores, co-tenant dev DB, tiny JSONB payloads, ~4–23% run-to-run variance. Single-point peaks (commit_delay, projector shards) sit within overlapping noise bands.

Specific caveats

  • No true disk spill was reached. The 10M-row "spill" tier (~1.5GB) still fit the 8GB container cache, so that degradation is index/table-size, not disk I/O. Raising OrbStack RAM and pushing a table past 8GB is the obvious next step.
  • Read parity is conservative, not identical. d4/d5 do a per-event heap read; d1/d2/d3 rewrite the delivery row but don't separately re-read the event payload — i.e. the comparison is tilted slightly against the no-fanout headline, so the real gap is at least what's shown.
  • Tiny payloads. Larger event bodies would make the per-event read cost more, narrowing d4/d5's lead somewhat (and making batching even more important).
  • Unexhausted lever: sharding the d4/d5 per-group frontier would lift the single-group wall (~100–128k) — not measured here.
Update: the last three caveats were directly tested in the second audit (§13) — read parity is fair (the gap is real, not a read-skipping artifact), frontier sharding lifts the g1 wall ~3.8×, and per-row pop-delete doubles the drain. True disk spill remains the one unmeasured lever.

13 · Second audit — corrections & bound-moving optimizations

A second independent pass: six parallel adversarial reviewers (read parity, unit fairness, breaking-point method, environment, bound classification, statistics) over code + raw data, plus a serial empirical re-test suite on a fresh throwaway PG18 container with the co-tenant dev DB verified idle throughout. Two questions: (1) did each technique measure what we claimed, and (2) which untried levers move the bounds. All re-tests are median-of-3.

One-line outcome: the directional findings and matched ratios hold and are now stronger (read parity tested; invariants machine-enforced) — but several "walls" are removable: per-row 124k → 258k with pop-delete, claim-log g1 → 521k by sharding the frontier; and the e2e 75k/287k/300k figures are burndown rates, not steady-state ceilings.

13.2 Read parity is fair — the gap is not a read-skipping artifact

The sharpest construct-validity worry was that per-row consumers never read the event body while claim-log does count(payload). Tested directly (units/s, g10/b200/cc16):

payloadper-row no-readper-row +readclaim-log +readfair ratio
tiny {"x":1}127.7k126.2k460.9k3.65×
~1 KB—131.9k560.2k4.25×
~4 KB—122.4k551.9k4.51×
Reading it: adding a payload read to the per-row consumer changes nothing (126k vs 128k) — the path is write-bound. Claim-log is flat across cached payload sizes because its batched sequential read is nearly free until reads leave RAM. So the no-fanout advantage is real, and "bigger payloads narrow it" only holds off-cache.

13.4 Breaking the per-row wall

The 2-UPDATE state-flip is not the only per-row option (units/s, g10/cc16, vs baseline):

variantb200b1000
baseline: ready→inflight→acked (2 UPDATEs)110–124k140k (+27%)
UNLOGGED deliveries (2-UPDATE)160k (+45%)—

Why: the wall is write volume, not durability latency. sync=off (fsync only) doesn't help, but removing WAL entirely (UNLOGGED) does (+45%), and replacing two row-rewrites with one delete helps most (+134%). pop-delete trades retry/lifecycle for speed (at-most-once on crash); its low WAL volume is why UNLOGGED adds nothing on top of it.

13.5 What kind of bound is each "wall"?

  • Closest to Postgres-fundamental: the UNLOGGED ~1.4–1.45M insert ceiling (CPU), and the architectural fact that a per-(group,event) row write costs >0 on both sides — the reason no-fanout wins.
  • This-hardware (8-core / cache), moves with more cores or RAM: the per-row 110–124k level, the d1 0.5M fanout plateau, the 666k projector peak, all e2e absolutes, and every "spill" number (cache-resident).
  • Harness artifact (removable now): the g1 frontier wall (→ shard, +280%); the per-row 124k as a write-pattern choice (→ pop-delete, +134%); the e2e "ceilings" (diverging burndown); and the read-parity concern (tested ≈ no effect when cached).
